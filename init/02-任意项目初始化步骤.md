# 任意项目初始化步骤

这份步骤适合新项目，也适合老项目补装 OpenSpec。核心原则只有一句话：先官方初始化，再叠加项目治理规则。

## 步骤 1：运行官方初始化

在目标项目根目录执行：

```bash
openspec init --tools codex
```

如果不是 Codex，而是别的 AI 工具，把 `codex` 换成对应工具名称即可。

## 步骤 2：保留官方骨架，不要先魔改

初始化后，优先保留这些官方产物：

- `openspec/config.yaml`
- `openspec/changes/`
- `openspec/specs/`
- `openspec/changes/archive/`
- 工具侧自动生成的 OpenSpec skills / commands

不要先把它们替换成自定义目录名，也不要一上来重写官方 prompt 分类。

## 步骤 3：补项目上下文

把 `templates/openspec-config.yaml.example` 合并到目标项目的 `openspec/config.yaml`，至少补齐这些内容：

- 项目类型
- 技术栈
- 主入口或主流程
- 关键目录职责
- 复用优先入口
- coding / apply 阶段要显式使用的技能
- 输入校验、安全、测试、回滚等边界

如果目标项目默认不是中文，把语言相关字段改成对应语言即可。

## 步骤 4：补根级长期规则

把 `templates/AGENTS.template.md` 合并到目标项目根 `AGENTS.md`。建议只保留长期有效规则，例如：

- 哪些场景必须走 OpenSpec
- 哪些设计改动或代码改动一律不能绕过 OpenSpec
- 是否需要确认闸门
- 目录职责
- 主入口兼容约束
- coding / apply 阶段显式使用哪些 Superpowers 技能
- 测试与验证要求
- 归档前知识沉淀要求

不要把一次性任务背景、临时 bug 现象、短期试验结论写进去。

## 步骤 5：为每个包级目录补自己的 `AGENTS.md`

不要只写根 `AGENTS.md`。初始化时，应该为每个包级、领域级、模块级目录建立自己的 `AGENTS.md`，至少覆盖：

- 本目录 / 本包职责
- 对外暴露的接口或能力边界
- 优先复用的本地封装
- 本目录特有的命名、测试、异常处理规则
- 哪些规则要比根目录更严格

建议直接基于 `templates/PACKAGE_AGENTS.template.md` 为每个包生成一份。

推荐优先覆盖这些目录：

- 应用入口目录
- 业务域目录
- 基础设施目录
- 数据访问目录
- 测试目录
- 资源或脚本目录

## 步骤 6：如果项目治理较重，再补项目手册

如果目标项目也需要像本仓库一样有更严格的 OpenSpec 手册，可新增：

- `openspec/CODEX_MANUAL.md`

建议直接基于 `templates/CODEX_MANUAL.template.md` 改。

即使项目很轻量，也建议至少保留“根 `AGENTS.md` + 包级 `AGENTS.md`”；是否需要单独手册再按治理强度决定。

## 步骤 7：补 review 治理，但不要伪装成官方命令

本项目之所以稳定，一个关键原因是：

- `apply` 完成后，不直接把 change 当成结束。
- coding / apply 阶段显式使用 `superpowers:test-driven-development`、`superpowers:requesting-code-review`、`superpowers:verification-before-completion`。
- 自动进入独立 `review`。
- `review` 完成后先停下，等待用户确认。
- 用户确认后，再考虑 `archive`。

这里要注意，`review` 是项目治理补充阶段，不是替代官方 `archive` 的新主命令。你可以：

- 在 `tasks.md` 里预留 review 任务；
- 或者在 change 目录下新增 `review.md`；
- 或者在根 / 包级 `AGENTS.md` 与手册里明确“apply 后自动 review，review 后等待确认”。

## 步骤 8：按阶段使用提示词

后续日常协作时，直接使用 `prompts/` 目录里的提示词：

- 初始化类
- 探索与提案类
- 实现与复核类
- 归档与沉淀类

这样可以保持不同项目里的 OpenSpec 对话风格一致。

## 不同项目类型需要替换什么

| 项目类型 | 重点替换内容 |
| --- | --- |
| Web / API | 路由边界、数据模型、接口兼容、鉴权、安全验证 |
| Python 自动化 | 主入口顺序、设备依赖、外部脚本封装、异常恢复 |
| CLI / 桌面工具 | 启动入口、配置文件、用户数据迁移、离线能力 |
| 数据处理 / ETL | 输入来源、幂等性、重跑策略、持久化边界 |
| SDK / 库 | 公共 API 稳定性、向后兼容、示例与文档同步 |

## 最后检查

完成初始化后，至少确认以下几点：

- 仍在使用官方 `openspec/` 目录结构。
- `openspec/config.yaml` 已填入项目上下文。
- 根 `AGENTS.md` 只承载全局规则，不混入包内细节。
- 根 `AGENTS.md` 已明确“设计改动和代码改动默认必须先走 OpenSpec”。
- 每个包级、领域级目录都已经补了自己的 `AGENTS.md`。
- 提示词分类清晰，成员知道什么时候用哪一类。
- coding / apply 阶段已经显式接入所需的 Superpowers 技能。
- 如果项目需要更高治理，已经明确“`apply` 自动 review，review 后等待确认”的位置和要求。
